# DEX vs Temporal vs LangGraph

This document compares DEX against [Temporal](https://docs.temporal.io/)
and [LangGraph/LangGraph-API](https://docs.superdurable.com/langsmith/agent-server) across the
two dimensions that matter most when picking a durable workflow / agent
orchestration platform: end-user feature surface and underlying system
architecture.

## Legend

The status emoji at the start of each cell gives a glanceable verdict, followed
by a short note:

- ✅ — supported / good
- ❌ — not supported / missing
- ⚠️ — partial, has caveats, or requires non-trivial workarounds
- ℹ️ — neutral fact (e.g. implementation detail, no value judgment)

## 1. User Feature Comparison


| Name                                                                                         | DEX                                                                                              | Temporal                                                                                                                            | LangGraph                                                                                                                                                                  | Why it matters                                                                                                                                                                                                |
| -------------------------------------------------------------------------------------------- | ----------------------------------------------------------------------------------------------------- | ----------------------------------------------------------------------------------------------------------------------------------- | -------------------------------------------------------------------------------------------------------------------------------------------------------------------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| 1.1 Step input limit                                                                         | ✅ All steps have access to workflow data "for free"                                                   | ⚠️ User has to select data from workflow data and serialize. 4MB limit, accounted into workflow history                             | ✅ All steps have access to workflow data "for free"                                                                                                                        | Allowing a step to have full context for LLM and tool calls without an extra storage system and IO                                                                                                            |
| 1.2 History size scalability                                                                 | ✅ No scalability limit                                                                                | ⚠️ Total history size should be <50MB and must be <200MB, including all input & output. `ContinueAsNew` is required as a workaround | ✅ No scalability limit                                                                                                                                                     | Allowing a workflow to run for as many steps as needed                                                                                                                                                        |
| 1.3 Workflow data size limit                                                                 | ✅ Predictable workflow data size, no replay constraint                                                | ⚠️ Unpredictable size limit, due to the replay constraint                                                                           | ✅ Predictable workflow data size, no replay constraint                                                                                                                     | Same as above — allowing a step to have full context for LLM and tool calls without extra storage system and IO                                                                                               |
| 1.4 Built-in streaming support                                                               | ✅ Yes                                                                                                 | ❌ No                                                                                                                                | ✅ Yes                                                                                                                                                                      | Support streaming intermediate output from LLM / tool call                                                                                                                                                    |
| 1.5 Step durability                                                                          | ✅ Configurable per step                                                                               | ❌ Not configurable                                                                                                                  | ⚠️ Configurable per graph / run (not per step)                                                                                                                             | Flexible for steps that need less durability at lower cost                                                                                                                                                    |
| 1.6 Versioning                                                                               | ✅ Simple and natural like iWF, with additional support to find waiting steps on old versions          | ⚠️ Complicated, especially for long-running workflows                                                                               | ⚠️ Could be simple, but missing documentation today; resuming a node from interrupt re-executes some code in the node                                                      | Code is changing all the time. It's critical for users to make code changes safely for reliability                                                                                                            |
| 1.7 Step timeout & heartbeat timeout                                                         | ✅ Yes — defaults to 5 minutes with 10s heartbeat timeout                                              | ⚠️ Yes — defaults to 30s; heartbeat timeout recommended for long-running steps but not enabled by default                           | ❌ No support for step timeout or heartbeat timeout                                                                                                                         | User code is not reliable. This is a must-have to be able to retry reliably without lots of custom code for every IO call in the step                                                                         |
| 1.8 Step retry policy & post-retry error handling                                            | ✅ Defaults to infinite retry, with durable backoff, and supports error handling after retry exhausted | ✅ Defaults to infinite retry, with durable backoff, and supports error handling after retry exhausted                               | ❌ Only local (non-durable) backoff retry; not enabled by default                                                                                                           | Must-have for a reliable process                                                                                                                                                                              |
| 1.9 Observability: last failure (stacktrace) on retry                                        | ✅ Yes                                                                                                 | ✅ Yes                                                                                                                               | ❌ No                                                                                                                                                                       | Proven to be super helpful for production troubleshooting & debugging                                                                                                                                         |
| 1.10 Observability: find runs failing/retrying on steps                                      | ✅ Yes                                                                                                 | ❌ No                                                                                                                                | ❌ No                                                                                                                                                                       | Highly requested feature for production troubleshooting & debugging among Temporal / iWF users                                                                                                                |
| 1.11 Observability: complex AND/OR/NOT search                                                | ✅ Yes                                                                                                 | ✅ Yes                                                                                                                               | ❌ No — only `AND` is supported in metadata filtering                                                                                                                       | Super helpful for managing millions of runs in production                                                                                                                                                     |
| 1.12 Durable timer & skip-timer API                                                          | ✅ Yes, with skip-timer API built-in                                                                   | ⚠️ Yes, but no skip-timer API built-in (needs custom code)                                                                          | ❌ No                                                                                                                                                                       | Must-have for building reliable long-running processes                                                                                                                                                        |
| 1.13 Async nodes (multi-threaded) in a run, with cancel/stop other threads                   | ✅ Yes                                                                                                 | ✅ Yes                                                                                                                               | ❌ No                                                                                                                                                                       | Agents are evolving into multi-agent coordination architectures to avoid being blocked on a single LLM call (e.g. start two LLMs, cancel the slower one once the first has a good answer or after user input) |
| 1.14 Async channels for external & internal communication, with `anyOf` / `allOf` conditions | ✅ Yes — both use the same channel concept                                                             | ⚠️ Yes — signal for external; internal queue uses a different mechanism                                                             | ❌ No — or partially via interrupt, which only pauses the whole run                                                                                                         | Critical mechanism for external and internal communication to build multi-threaded processes; even more useful combined with durable timers                                                                   |
| 1.15 Run hard timeout & soft (graceful) timeout                                              | ✅ Yes for both                                                                                        | ⚠️ Yes for hard timeout; needs complex user code for soft timeout                                                                   | ❌ Only deployment-level timeout for all runs                                                                                                                               | Timeout is important to ensure agent runs stop as expected                                                                                                                                                    |
| 1.16 Fan-out execution with controlled parallelism                                           | ✅ Yes via `ChildWorkflow` for unbounded fan-out                                                       | ✅ Yes via `ChildWorkflow` for unbounded fan-out                                                                                     | ❌ No — `SubGraph` is not designed for fan-out (only runs within the same run)                                                                                              | Some cases need to fan out to runs with more CPU/memory resources (so multiple parallel steps in one run won't work), and need to control parallelism reliably                                                |
| 1.17 Learning curve                                                                          | ✅ Very simple to learn and use, with minimal but powerful concepts                                    | ⚠️ Known for a steep learning curve                                                                                                 | ⚠️ Pretty simple at the surface, but has many (some immature) concepts for advanced use cases: Functional API like Temporal, `Send` (especially from `SubGraph` to parent) | The lower the learning curve, the faster teams can adopt it and the fewer foot-guns in production                                                                                                             |


## 2. Architecture Comparison


| Name                                                 | DEX                                                                            | Temporal                                                                                            | LangGraph                                                                                                                                                     | Why it matters                                                                                                                                                                                                                                                          |
| ---------------------------------------------------- | ----------------------------------------------------------------------------------- | --------------------------------------------------------------------------------------------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------- | ----------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| 2.1 Built-in REST/HTTP agent server for frontend use | ❌ No                                                                                | ❌ No                                                                                                | ✅ Yes                                                                                                                                                         | Definitely useful, but DEX stays focused on orchestration and lets a higher-level abstraction deal with this problem                                                                                                                                               |
| 2.2 Scalability: run & core orchestration            | ✅ Linear scalability with sharded DB, upgradable max shards, fully distributed      | ✅ Linear scalability with sharded DB, fixed max shards (until cluster migration), fully distributed | ❌ Limited in Postgres; a single run queue with a single-threaded sweep and global locks; not fully distributed                                                | Core throughput ceiling for the whole platform                                                                                                                                                                                                                          |
| 2.4 Scalability: checkpointing                       | ✅ Distributed & sharded by `runID`, with blob store (e.g. S3) for large data        | ✅ Distributed & sharded by `runID`, with blob store (e.g. S3) for large data                        | ⚠️ Currently no, but working on moving to sharded MongoDB + blob store (e.g. S3) for large data                                                               | Checkpoint write throughput limits how fast workflows can advance                                                                                                                                                                                                       |
| 2.5 Scalability: searching                           | ✅ Separate from core storage, can use ElasticSearch for scalability                 | ✅ Separate from core storage, can use ElasticSearch for scalability                                 | ❌ Limited inside the current core run queues; searching draws a lot of resources and slows down core capability                                               | Search workloads must not degrade the orchestration hot path                                                                                                                                                                                                            |
| 2.6 Multi-tenancy                                    | ✅ Yes                                                                               | ✅ Yes                                                                                               | ❌ No                                                                                                                                                          | Multi-tenancy is the best way to build a healthy business with good margin and low prices                                                                                                                                                                               |
| 2.7 Main implementation language                     | ℹ️ Go in server, with a very thin SDK communicating via gRPC                        | ℹ️ Go in server, with a fat SDK communicating via gRPC (using a Rust core to share most SDK code)   | ℹ️ Python in `langgraph-api`, with most code in the SDK                                                                                                       | Go has proven to be the best language for this kind of system. Python is good for prototypes but not for building large-scale production systems                                                                                                                        |
| 2.8 Service API mode for worker task dispatch        | ✅ Polling for tasks/runs over gRPC API                                              | ✅ Polling for tasks/runs over gRPC API                                                              | ❌ Worker & server are the same process; in DR they tried push mode                                                                                            | Polling lets customers (especially enterprise) host their code anywhere outside our network and still use the service. It's also more efficient: when no worker is available, the server doesn't keep busy pushing with backoff — the task is persisted for next pickup |
| 2.9 Consistent APIs across OSS and Cloud             | ✅ Yes — OSS will have the same APIs & SDKs; most implementation is in the same repo | ✅ Yes — OSS has the same APIs & SDKs; most implementation is in the same repo                       | ❌ No — OSS has to use different APIs; implementation is split between internal and OSS repos                                                                  | Consistency between OSS and Cloud reduces friction for users adopting either path and lowers maintenance cost                                                                                                                                                           |
| 2.10 System operation & observability                | ✅ Will have good observability from the beginning                                   | ✅ Same — strong operational observability                                                           | ⚠️ Was poor and is being improved, but improvements are difficult because of the split between two repos and the blurred line between server code vs SDK code | Operability determines how reliably the platform itself can be run in production                                                                                                                                                                                        |
| 2.11 Tech debt                                       | ✅ Will not have tech debt (greenfield)                                              | ℹ️ Doesn't matter to us in this comparison                                                          | ⚠️ Tech debt around `RunnableConfig`, plus many legacy streaming APIs / schemas                                                                               | Tech debt directly impacts iteration speed and reliability of new features                                                                                                                                                                                              |
